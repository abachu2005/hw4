Q1: FoundationDB decouples its Transaction Management System from its Storage System so that the two layers can be scaled, recovered, and reasoned about independently. The TMS (Sequencer, Proxies, Resolvers, and LogServers) is responsible for assigning versions, ordering writes, detecting conflicts, and persisting the redo log, while the SS is a horizontally sharded layer of StorageServers that holds the actual key-value data and serves reads. This split lets FDB push transactional state into an in-memory tier that can be rebuilt cheaply, while keeping bulk data on durable storage that does not have to participate in commit-time coordination.

Q2: The Resolvers are responsible for detecting read-write conflicts between concurrent transactions. Each transaction sends its read set and write set to the Proxies, which forward them to the Resolvers; this happens during commit, after the Sequencer has issued a commit version but before the writes are durably appended to the LogServers. A Resolver checks whether any committed transaction in the range (read_version, commit_version] wrote a key that overlaps the current transaction's read set, and if so the transaction is aborted, otherwise it is allowed to proceed to logging.

Q3: The read version is an MVCC timestamp handed out by the Sequencer that pins the transaction to a globally consistent snapshot of the database as of that version. All reads within the transaction observe the same snapshot, which is what gives FDB its serializable / external-consistency guarantee: because commit versions are also issued by the same Sequencer in monotonic order, the system can reason about strict ordering between transactions and reject any transaction whose snapshot would be invalidated by a later commit.

Q4: A snapshot read does not contribute its key to the transaction's read set, so the Resolver never checks it for read-write conflicts at commit time. A standard read always adds the key to the read set, meaning a concurrent write to that key forces an abort. You would prefer a snapshot read when you only need a consistent point-in-time view and do not care about being invalidated by a concurrent writer, and a standard read when the transaction's correctness depends on the value not changing under it.

Q5: Recovery is cheap because StorageServers are continuously pulling and applying mutations from the LogServers as part of normal operation, so the durable on-disk state of each StorageServer is already very close to the latest committed version. When the TMS fails, only the small TMS tier has to be reconstituted, as the new TMS recovers by reading the tail of the previous LogServers up to the last durable version, then resuming. Because the SS already has the data on disk and was tailing the log, there is no big redo in the critical path, and reads can resume against the StorageServers almost immediately.
